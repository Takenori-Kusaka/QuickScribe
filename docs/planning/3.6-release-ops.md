# 3.6 リリース / 運用 / バージョニング計画

> Status: DRAFT (2026-06-16) — ADR-0007準拠（意思決定駆動・反証可能）。一次情報ベース。
> [ADR-0005](../adr/0005-tech-stack.md)（配布・署名・updater既決）を具体化し、次の **CI/CD ADR** の入力とする。

## 採用結論（answer-first）

| 論点 | 結論 | 根拠 |
|---|---|---|
| バージョニング | **0.x を当面維持**（Logseq/Neovim方式）。タグ `v0.MINOR.PATCH`、プレリリース `v0.x.y-beta.N`（ドット+数値） | SemVer 2.0 条項4(0.xは随時変更可)・条項11(数値比較) |
| チャネル | **stable / nightly の二系統**（VS Code Insiders型） | 多系統は単独開発に過剰 |
| 変更管理 | **release-please**（Release PR方式） | Rust+Node混在を1ツールで賄える唯一の現実解 |
| GitHub Releases | **tauri-action集約**、annotated tag、`make_latest=true`明示、`attest-build-provenance`(Sigstore)+`SHA256SUMS` | REST/Releases仕様 |
| 配布(Win) | **SSL.com eSigner(クラウドIV/OV署名 約$20/月)+winget** | Azure Artifact Signingは日本・個人が対象外 |
| 配布(Linux) | **AppImage+.deb維持+Tauri updater**、Flathubは到達拡大候補(portal権限が条件) | 配布形態比較 |
| サポート | **最新版のみ・ベストエフォート**（Rust型、自動更新前提） | LTSは過大 |

## 重要発見（意思決定を変える）

1. **Windows署名の分岐点＝「日本・個人」**: Microsoft Azure Artifact Signing（$10/月・トークン不要・CI直結）は**個人はUS/Canadaのみ、日本は対象外**。→ 現実解は **SSL.com eSigner**（HSM不要クラウド署名・CI直結・個人IV取得可・国際提供）。約$20/月、30日無料。2023年6月以降OV/EVは鍵エクスポート不能(HSM内生成)が必須で「鍵ファイルでCI署名」は消滅。
2. **「署名すれば警告ゼロ」は誤り**: SmartScreenは①発行者reputation②ファイルhash reputationの2軸。**署名済みでも新バイナリはreputation蓄積まで警告**。**EVはもはやSmartScreenをバイパスしない**（Microsoft明言）→ **EVに割増を払う理由なし**。完全回避はMicrosoft Storeのみ。
   - → [ADR-0005](../adr/0005-tech-stack.md) Accept前ゲート3「SmartScreen警告なし」を「①署名が正しく検証②発行者名表示③reputation蓄積で警告が消える経路」に**精緻化**する。
3. **Tauri updater鍵はSPOF**: minisign署名は必須・無効化不可。**鍵喪失=既存ユーザーへの更新が永久断絶**。out-of-boxの鍵ローテーション無し（シーケンス法のみ）。公式advisory GHSA-2rcp-jvr4-r259: `envPrefix`に`TAURI_`を含むと秘密鍵がフロントに混入しうる → **`envPrefix:['VITE_']`固定＋CIで`dist/`に鍵混入grep検証**。

## 自動化パイプライン（CI/CD ADRへ）

1. commit: Conventional Commits徹底＋**squash-mergeデフォルト**。
2. **release-please**: Cargo.toml(native)＋`tauri.conf.json`(extra-files)版同期。Release PR=リリース承認ゲート。0.x期は`BREAKING→MINOR`丸めを設定。
3. Release PRマージ→**tauri-action**: Win(NSIS/MSI署名済)＋Linux(AppImage/.deb)、`includeUpdaterJson:true`で`latest.json`+`.sig`（plugin-updater≥2.10.0）。
4. 署名: updater=minisign(Secrets, envPrefix固定+grep)、Win Authenticode=**SSL.com eSigner**をCIから。
5. provenance/checksum: `actions/attest-build-provenance`+`SHA256SUMS`。
6. 配布登録: winget(`wingetcreate`自動PR)、Linux=AppImage/.deb直+updater、（条件付き）Flathub。
7. セキュリティゲート: **cargo-audit + cargo-deny + npm audit + Dependabot** 必須。

## チャネル/更新運用
- stable: タグ`v0.x.y`→正式Release(`make_latest=true`)、updaterは`releases/latest/.../latest.json`固定URL。
- nightly: mainプッシュ/cron→prerelease、別`latest.json`/別エンドポイント、**署名鍵をstableと分離**（被害局所化）。
- 鍵Runbook: 複数バックアップ＋シーケンス法ローテーション手順を事前文書化。

## サポート/セキュリティポリシー（SECURITY.md方針）
- 最新版のみ・ベストエフォート（自動更新が全ユーザーを最新化）。
- 報告は**GitHub Private Vulnerability Reporting**（公開issue不可）。修正版を自動配布後にGHSA公開（coordinated disclosure）。
- ローカルデータ/設定は**スキーマ版＋起動時自動migration＋非破壊**（事実上のpublic API）。
- 脆弱性報告で録音/文字起こしデータを送らせない旨を明記。

## 反証条件（覆る場合）
- Azure Artifact Signingが日本/個人へ拡大→$10/月へ乗換。
- Microsoft Store(MSIX)が常駐/WASAPIループバック/ホットキー/物理トリガーと両立→Storeが摩擦ゼロ最短。
- Flatpak portalで常駐機能が成立しなければFlathub断念。
- 法人/有償顧客が付けば明示SLA（ベストエフォート撤回）。

## 未解決（実機/契約で確定）
1. SSL.com eSigner IVを日本居住個人が取得しGH Actionsから署名できるか（本人確認・実コスト）。
2. Microsoft Store(MSIX)が常駐/ループバック/ホットキー/物理トリガーと両立するか。
3. Tauri(WebKitGTK)のFlatpak portalでホットキー/音声/ループバックが成立するか（ADR-0005 Linuxスパイクに統合）。
4. `tauri.conf.json`のversion省略→Cargo.tomlフォールバックが効くか、release-please extra-filesマーカー位置。
5. updaterのSemVer比較が「stable > nightly先行版」を誤判定しないか。
