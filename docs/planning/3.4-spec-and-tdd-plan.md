# 3.4 仕様策定フェーズ計画（仕様駆動 × TDD t-wada）

> Status: DRAFT (2026-06-16)
> 根拠: [spec-driven-development.md](../research/spec-driven-development.md) / [driven-development.md](../research/driven-development.md) / [ADR-0007](../adr/0007-research-question-framing-method.md)

## 方針（再掲・確定事項）

[spec-driven-development.md](../research/spec-driven-development.md) の結論を運用に落とす:
- **要件(WHAT)と設計(HOW)を分離**（ISO/IEC/IEEE 29148 "what not how"。JIS X 0166準拠）。
- **軽量BDD中心ハイブリッド**: 受入条件は Given-When-Then を主、**EARSは固まった機能のみ**。Kiro流の Requirements/Design/Tasks 章立てを踏襲。重いSpec Kitフル儀式は初期不採用。
- **GOOS二重ループ**: 外側=受入テスト / 内側=TDD(t-wada Canon)。**ユビキタス言語を仕様・テスト名・モデルに貫通**。Living Documentation（仕様=テスト）で形骸化防止。

## 仕様の成果物構造（issue/機能ごと）

各 Story/機能ディレクトリ `specs/<feature>/` に:
- `requirements.md` — ユーザーストーリー＋受入基準。安定機能はEARS（`When <trigger>, the system shall <response>` / `If <unwanted>, then ...`）、探索的は自由記述＋BDD例。
- `design.md` — アーキ/コンポーネント/シーケンス/非機能（セキュリティ・性能・プライバシー）。
- `tasks.md` — 依存順の実装タスク（テスト含む）。
- 語彙は[ユビキタス言語辞書](3.3-personas-and-storybook.md)（DDD）と一致させる。

## 仕様 → テスト → 実装の標準サイクル（DoD）

```
ユビキタス言語確認
  → requirements(受入基準: BDD/EARS) を書く
  → 受入テスト(外側ループ)を作り Red を確認
  → Canon TDD(内側ループ): テストリスト→1つ→Green→Refactor（歩幅: 仮実装/三角測量/明白な実装）
  → DDDモデルへ知見を還流（用語変更は仕様・テスト・モデルを同時更新）
  → CI全green(単体+結合+E2E) → レビュー → マージ → 自動リリース
```

## TDD(t-wada)運用規律（形骸化防止）

- テストリストを先に書く（Canon TDDの前段）。テストは1つずつ、Redを実際に確認。
- 偽りのGreen禁止（アサーション削除・期待値コピペ禁止）。Refactor局面を省かない。
- TDDは設計手法として使う（自動テストは副産物）。
- **AIエージェント運用契約**: 「受入テストを先に書き、Redで失敗を確認してから内側ループ」を義務化し、AIの完了詐称を機械的に止める（Fowler「AIは100%遵守しない」留保への対策）。

## テスト観点の網羅（ステップ5で各issueに適用）

単体 / 結合 / システム / E2E（tauri-driver+WebdriverIO実起動）/ UI-UX / ストーリーブック整合 / セキュリティ / 非機能（性能・プライバシー）。詳細は[3.5 全体テスト計画](3.5-waterfall-and-quality.md)。

## このフェーズの完了条件
- 仕様テンプレ（requirements/design/tasks）と受入基準記法（BDD既定＋EARS）を確定。
- ユビキタス言語辞書の初版（3.3）と接続。
- ステップ4のissueに「仕様＋テスト設計を先に」を組み込める状態にする。
