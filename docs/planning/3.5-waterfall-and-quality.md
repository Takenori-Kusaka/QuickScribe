## テスト計画（観点 × レベル）

| レベル | 範囲 | 手段 |
|---|---|---|
| 単体 | Rust関数/Svelteロジック | `cargo test` / `vitest` |
| 結合 | コマンド↔フロント、エンジン差し替え境界 | Rust統合テスト / モック |
| システム | アプリ全体の振る舞い | シナリオテスト |
| **E2E** | 実起動してUI操作 | **tauri-driver + WebdriverIO（Linux+xvfb、CI実起動）** |
| UI/UX | アクセシビリティ/操作性 | チェックリスト＋手動 |
| ストーリーブック整合 | ペルソナ別シナリオの成立 | [3.3](3.3-personas-and-storybook.md)を実行可能化 |
| セキュリティ | 秘密情報/入力検証/権限最小 | 静的解析(CodeQL)/依存(Dependabot)/手動レビュー |
| 非機能 | 性能(起動/推論)・プライバシー(ローカル完結)・信頼性 | ベンチ/監査 |

CIゲート: 単体＋結合＋E2E＋型＋lint＋セキュリティスキャンを必須化（[テスト基盤タスク](../../)）。

## 品質規格の準拠範囲（定義）

- **ISO/IEC 25010（製品品質モデル）** を品質特性の基準枠として採用し、本プロジェクトで**重点的に保証する特性**を明示:
  - 機能適合性 / 信頼性 / 使用性 / **セキュリティ** / 互換性(Win/Linux) / 保守性 / 移植性 / 性能効率。
  - 重点: **セキュリティ・使用性（簡便さ）・信頼性**（コア価値と直結）。
- **準拠の現実的範囲（正直な定義）**: 単独OSSのため、規格の**完全な認証取得は対象外**。25010を「品質特性の網羅チェックリスト」として用い、各特性に受入基準とテストを割り当てる**自己評価準拠**を目標とする。
- セキュリティは OWASP（該当する範囲）と最小権限・秘密情報管理（[.env/keyring方針](../adr/0005-tech-stack.md)）を基準に。
- アクセシビリティは WCAG を参照（UIの色コントラスト/キーボード操作）。
