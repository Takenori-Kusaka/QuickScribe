## 1. 主要ブロッカー（v1.0前に解消すべき）

### コード/UX（実害）
- **a11y(item6)**: 設定/ジャーナルのモーダルに `role=dialog`/`aria-modal`/フォーカストラップ/Esc閉じが無く、SR・キーボードユーザーが安全に操作不能。コントラスト不足（`.hint`/`.restyle-label` #9ca3af=2.54:1）。
- **UI文言(item9)**: `App.svelte:1660` に内部ID **「モデル（S2.2）」** が露出。raw error（`String(e)`/`${e}`）を画面に直表示。
- **入力制約(item8)**: `transcribe_file`(lib.rs:408) に**ファイルサイズ/長さ上限のガードも通知も無い**。巨大入力で破綻リスク。

### リリース/ガバナンス
- **リリースノート(item11)**: 全19リリースが同一定型文。変更点が読めない → release-please等(#48)。
- **ブランチ保護(item17)**: 未設定(404)。CLAUDE.md「mainは常にリリース可能」と矛盾。main直push可能。
- **コード署名(item7/E8 #50)**: 全リリース未署名(SmartScreen警告)。SignPath Foundation申請=**審査待ち**（PR#386にCI組込ドラフト）。

### 大規模（基盤なし＝1.0必須なら早期着手必須）
- **i18n多言語(item12)**: フレームワーク未導入・日本語ハードコード(フロント約150＋Rust約200＝約300-350文字列)。en/zh/es ゼロ。**Rustエラーのコード化**が必要。工数13-20人日。
- **テストカバレッジ(item13)**: 計測基盤ゼロ(vitest coverage/llvm-cov無)。App.svelte 2508行未テスト。80%は証明不能。工数6-10人日。
- **非機能ベンチ(item14)**: perfベンチ/CI/精度ベンチ完全不在。工数7-11人日。

## 2. 良好な点（v1.0の土台）
- **セキュリティ(item2)**: 配布バイナリに実害脆弱性なし。npm audit 10件は**全てdevDependencies**（ランタイム無影響）。API鍵/AWS鍵ともkeyring化。既定STT=ローカル・テレメトリ無し（コードと一致）。updater minisign署名・最小capabilities・XSSシンク0。
- **コード品質(item3/4)**: Rustはtrait抽象(TranscriptionEngine/FormattingEngine)・純粋関数分離・テスト58＋統合1・rustdoc高密度で芯が通る。
- **OSSコミュニティ(item17)**: 健全度100%（README/LICENSE/CoC/CONTRIBUTING/SECURITY/Issue・PRテンプレ/dependabot.yml）。topics/description/homepage/Discussions整備。
- **ドキュメント体系(item1)**: ADR17本・仕様・調査(原典保存)が充実。差別化メッセージは利用者・貢献者向けに一貫。

## 4. v1.0.0 カットライン（要Decider判断・新ADR）
全観点の完全達成は概算**40人日超**。スコープ規律上、独断のフェーズ送りは不可。以下の論点をDecidersが決定し新ADRに記録する:
- **A案: フル達成後にv1.0.0**（i18n全4言語・カバレッジ80%・perf CI・a11y AA・署名すべて完了。最も堅いが数週間）。
- **B案: コア体験＋公開品質を満たす"1.0最小線"を定義**し、大規模ストリーム（i18n多言語・80%カバレッジ・perfベンチ）は**1.0で基盤導入＋一部、残りをv1.x段階**に（ADRで明示。スコープ縮小ではなく順序付け）。
- いずれでも **v1.0前必須**: a11uブロッカー(WS-A)・UX文言ブロッカー(WS-B)・入力制約(WS-C一部)・リリースノート(WS-J)・ブランチ保護(WS-K)・ドキュメント矛盾解消(WS-I一部)・README実スクショ(WS-N)。
