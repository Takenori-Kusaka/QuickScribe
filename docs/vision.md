# QuickScribe — Product Vision

> Status: Draft v0.1 (2026-06-15) — 企画フェーズの源流文書。以降の全計画・仕様・ペルソナ・テストはここから派生する。

## ポジショニング（一行）

**思考整理・自己理解のための、ローカル完結ボイスジャーナル。**
話した音声を、ニュアンスを残しつつ賢く成形・要約し、自分の考えを整理し自己理解を助ける。

## コア価値

文字起こし精度ではなく **「整形の知性」** が本体。逐語 ⇄ 要約 ⇄ ブレストを行き来できる。
文字起こしは差し替え可能なコモディティ入力に格下げする（[ADR-0002](adr/0002-stt-engine-strategy.md)）。

## 設計上の核心課題

**「リッチすぎると簡便でなくなる」バランス。** ここの解像度が勝敗を分ける。
ワンボタン起動 → 最小操作 → 整形済みメモが指定フォルダに着地、までの摩擦をゼロに保ちながら、
整形の知性を深める。

## 差別化に投資する4軸

1. 思考整理・自己理解への用途特化（競合の唯一の空白）
2. ニュアンス保持の整形思想（逐語スタイルを内蔵）
3. ローカル完結プライバシーの明確な約束（思考の生データを外に出さない）
4. 物理ボタン（Stream Deck / マウス / ホットキー）→ 整形済みメモ着地までの摩擦ゼロ統合体験

## 差別化軸にしないもの（コモディティ/競合追随済み・OSコモディティ化リスク）

- 文字起こし精度そのもの
- Win/Linux 対応そのもの
- ローカル動作そのもの

## 競合判定サマリ

条件付き GO。「技術の新しさ」で売るなら NO-GO（OpenWhispr 等が技術ピースを実装済み、
Windows 2026 がOS統合予定）。勝機は **用途特化 × 統合体験** にのみ存在する。
詳細: [competitive research](research/competitive-landscape.md)（後日転記）。

## 想定スコープ外（現時点）

- 会議議事録特化（Otter/Granola の土俵 — 用途が異なる）
- リアルタイム話者分離・字幕用途
- モバイルアプリ（初期は Win/Linux デスクトップに集中）

## オープン課題

- プロダクト名: "QuickScribe" は「素早い文字起こし」を想起させ、思考整理ジャーナルの本質とズレる。改名検討。
- 「整形の知性」をローカルLLMで完結させるか、BYO-API（Gemini/Claude）必須にするか（プライバシー約束との緊張）。
