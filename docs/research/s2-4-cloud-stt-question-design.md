## MECE サブ問い（各: 目的 / 出力 / ソース境界 / VOI / 反証）

### Q1. 各社の文字起こしAPIの最小仕様（Groq / Deepgram / OpenAI / Azure）
- 目的: バッチ（録音停止後に音声ファイルを投げる）文字起こしの **エンドポイント / 認証ヘッダ / 音声の渡し方（multipart or raw body）/ 対応フォーマット / 日本語指定 / レスポンスJSONのテキスト取り出し** を確定。
- 出力: プロバイダ別の最小リクエスト仕様表（URL / method / auth / body / 応答パス）。
- ソース境界: 各社公式APIリファレンス（一次情報）。
- VOI: どのプロバイダを初期対応にするか、共通化できる送信形式（WAV/mp3）を決める。
- 反証: いずれかが multipart 必須でureqでの実装が重い等なら、対応順を変える。

### Q2. 音声形式の送信
- 目的: 当方の16k mono f32 を各APIが受ける形式（WAV/FLAC/mp3/Ogg）へどう変換して送るか。既存 hound(WAV)/opus を再利用できるか。
- 出力: 送信前エンコードの方針（最小= 16k mono WAVバイト列をmultipartで送る等）。
- ソース境界: 各社「対応フォーマット」節。
- VOI: エンコード実装の追加要否。
- 反証: WAVが全社で通るなら追加エンコード不要。

### Q3. 同期(ureq)で完結するか / ストリーミングの要否
- 目的: 既存の同期HTTP(ureq)でバッチ文字起こしが完結するか。リアルタイムストリーミングはMVPで不要か。
- 出力: MVP=バッチのみで足りるかの判断（録音停止後一括が現UXと一致）。
- VOI: ureqで足りればtokio不要（AWSと同様の軽量方針）。
- 反証: バッチAPIが無くストリーミング必須の社があれば対応順を後回し。

### Q4. プライバシー/コスト/制約の明示
- 目的: 各社のデータ取り扱い（学習利用の有無/保持）・サイズ上限・料金の桁感を一次情報で押さえ、UIで正直に注意喚起できるようにする。
- 出力: 注意書きに使う事実（上限MB・データ利用方針）。
- VOI: 既定ローカルを正当化し、ユーザーの納得選択を支える。

## Step 8 収束（最重要3問）
Q1（API仕様）→ Q2（音声送信）→ Q3（同期で足りるか）。Q4は実装の注意書きに反映。

## メトリクス（GQM）
Goal: 公式最小APIで CloudEngine を安全に差し込む。
Metric: 各プロバイダに「公式一次情報での裏取り」「ureq同期で実装可否 Yes/No」「16k WAV送信で動くか」を記録。
