## Step 0 リフレーミング（5レンズ）

- 当初の問題: 「仮想デバイス不要でシステム音声(既定スピーカー)を録る」。
- **ユーザーによる再枠組み（2026-06-24）**: 「録音対象＝マイクの一種として選べるようにする。既定スピーカーだけでなく、任意のスピーカー(出力)デバイスも選択肢に出す」。
  - → S1.2 のデバイス選択UIに **出力デバイスのループバック** を統合する問題に変わる。「ループバック機能」ではなく「**録音ソースの抽象と列挙・選択の統一**」が真に解くべき問題。
- 「より良く解ける問題」: *入力(マイク)と出力(ループバック)を同一の `AudioCapture` ソース抽象＋同一の選択UIで扱うには、各OSのネイティブAPIをどう抽象化すべきか*。

## Step 1 QFocus（焦点文・疑問文にしない）

> VCチャット/録画OSSと競合は、仮想デバイス無しのシステム音声＋出力デバイス選択を、どのネイティブAPI・抽象・UIで実現しているか。QuickScribe が独自実装に陥らず prior art に倣うための設計判断材料。

## Step 2 ジョブ接続（JTBD）

> When 会議/通話/再生中の音声を後で振り返りたい時、I want to 相手側やPC再生音をマイクと同じ手軽さで録音ソースとして選びたい、so I can 自分の発話だけでなく対話全体を思考整理ジャーナルに残せる。

コア価値（思考整理）接続: 対話・会議の「相手の発話」を取り込めて初めて、振り返り対象が「自分の独白」から「対話全体」に広がる。

## Step 5+6 MECE サブ問い（各: 目的 / 出力形式 / ソース境界 / VOI=変える意思決定 / 反証条件）

### Q1. Windows: ネイティブAPIの選択
- **目的**: WASAPIループバックの2方式 — (a) レンダーエンドポイントの `IAudioClient`+`AUDCLNT_STREAMFLAGS_LOOPBACK`、(b) **Process Loopback API**（`ActivateAudioInterfaceAsync`+`AUDIOCLIENT_ACTIVATION_TYPE_PROCESS_LOOPBACK`）— を、OSS/競合が用途別にどう使い分けているか確定する。
- **出力**: 比較表（方式 / 取得単位=デバイス全体orプロセス / 最低Windowsバージョン / 無音時にデータが来るか / 採用OSS）。
- **ソース境界**: Microsoft Learn 公式 / OBS `win-capture`・`win-wasapi` 実コード / miniaudio・PortAudio 実装。一次情報のみ。
- **VOI**: 「デバイス全体ループバック方式(a)」を採用するか「プロセス単位(b)」も将来見据えるかの設計分岐を変える。出力デバイス選択UIには(a)が直結。
- **反証条件**: もし主要OSSが既定スピーカー録音にも(b)プロセスループバックを使っているなら、(a)前提の設計は見直す。

### Q2. Windows: 出力(レンダー)デバイスの列挙と選択
- **目的**: 「既定スピーカーだけでなく任意のスピーカーを選ぶ」を、`IMMDeviceEnumerator`(`eRender`)列挙＋選択デバイスでのループバック起動でどう実装するのが定石か。
- **出力**: 列挙→選択→キャプチャ起動の最小手順（擬似コード）と、競合UIの提示方法（例: Audacity の WASAPI ホスト＋"(loopback)"デバイス名、OBSの出力キャプチャ）。
- **ソース境界**: Microsoft Learn、Audacity/PortAudio、OBS。
- **VOI**: 設定UIで出力デバイスを別リストにするか、入力デバイスと統合した1リスト(種別バッジ付き)にするかを変える。
- **反証条件**: 競合が一貫して入力と出力を別UIに分けているなら、統合1リスト案を再考。

### Q3. Rust 実装手段（独自実装回避の核心）
- **目的**: 手書き windows-rs を避けられる既存Rustクレートの成熟度を確定。`cpal`(ループバック対応状況)、`wasapi` クレート、`pipewire-rs`、cross-platform の `miniaudio`(Rustバインディング) 等。
- **出力**: クレート比較（ループバック対応OS / 出力デバイス選択可否 / メンテ状況 / ライセンス[deny.toml整合] / 採用実績）。
- **ソース境界**: docs.rs / GitHub の一次情報、各READMEとissue。
- **VOI**: 「既存クレート採用」か「薄い自前FFIラッパ」かを変える。既存の単一クレートで全OSのループバック＋デバイス選択が賄えるなら自前実装をしない。
- **反証条件**: いずれのクレートも出力デバイス選択ループバックを安定提供しないと分かれば、OSごとFFIの段階実装に倒す。

### Q4. Linux: PipeWire / PulseAudio の monitor
- **目的**: PipeWire の monitor ソース取得が現行の定石か、PulseAudio monitor フォールバックは要るか。cpal/ALSA 経由で monitor が見えるか。
- **出力**: 取得経路（pipewire-rs vs libpulse vs cpal）と、出力デバイス(sink)単位の monitor 選択方法。
- **ソース境界**: PipeWire 公式 / OBS Linux 実装 / pipewire-rs。
- **VOI**: Linux 実装をどのライブラリに賭けるかを変える。
- **反証条件**: 主要ディストリで PipeWire 経由 monitor が cpal から普通に列挙できるなら、専用ライブラリ依存を避けられる。

### Q5. マイク＋システム音の同時取得とクロックずれ
- **目的**: 「自分の声＋相手の声」を両取りする場合の2デバイス同時キャプチャとサンプルレート/クロックドリフト処理を、OSSがどう扱うか（リサンプル・PLL・タイムスタンプ整合）。
- **出力**: 同時取得が必要か（MVPは単一ソース選択で足りるか）の判断と、必要時の定石。
- **ソース境界**: OBS（複数音声ソースのミキシング）、miniaudio。
- **VOI**: MVPスコープ — 「単一ソース選択のみ」か「mic+loopback同時」かを変える。**スコープ規律(ADR-0006)上、最終ゴールから落とさない**前提で、段階(フェーズ)実装の線引きに使う。
- **反証条件**: 単一ソース選択でもコアジョブ(対話の振り返り)が成立する具体例が多数なら、同時取得は後続フェーズへ分割（削除ではない）。

## Step 8 収束（最重要3問）

1. **Q3**（Rust実装手段）— 独自実装回避の可否を直接決める、最もイシュー度が高い。
2. **Q1**（Windows API方式）— 設計の土台。
3. **Q2/Q4**（出力デバイス列挙・選択）— ユーザー再枠組みの核心(任意スピーカー選択)。

## メトリクス（GQM）
- Goal: 独自実装に陥らず prior art に倣ったループバック＋出力デバイス選択を設計する。
- Question→Metric: 採用するAPI/クレートそれぞれに「≥2件の一次情報での裏取り」「ライセンスが deny.toml と整合」「出力デバイス選択を満たすか Yes/No」を必須項目として記録。
