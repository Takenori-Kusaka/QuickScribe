# S1.5 物理トリガー連携 — deep research 問い設計

> Status: Working (2026-06-24) / [ADR-0007](../adr/0007-research-question-framing-method.md) 準拠
> 対象 Issue: #21（Epic E1 #8）。差別化「物理ボタン統合体験」の核。

## Step 0/1 リフレーミング・QFocus

既存基盤: 録音トグルは (a) 設定可能なグローバルホットキー (b) CLI `quickscribe --toggle-record`（single-instanceで常駐へ転送）で起動できる。
→ S1.5の真の問題は「物理デバイス用の新APIを自作する」ことではなく、**既存の起動点（ホットキー/CLI）に物理デバイス（Stream Deck・多ボタンマウス・フットスイッチ等）をどう"正しく標準的に"橋渡しするか**。独自実装を避け各デバイスの公式手段に倣う。

## Step 2 ジョブ接続（JTBD）

When 思考が湧いた／会議が始まった瞬間に、I want to キーボードを探さず物理ボタン一押しで録音を開始/停止したい、so I can 思考の流れを途切れさせずに捉えられる（コア価値＝思考整理の即時性）。

## MECE サブ問い（各: 目的 / 出力 / ソース境界 / VOI / 反証）

### Q1. Stream Deck 公式プラグインの正道
- 目的: Stream Deck で外部アプリの録音トグルを叩く**公式に推奨される方法**を確定（公式プラグインSDK vs 既存の「Hotkey」「Open/System: Run」アクションで足りるか）。
- 出力: 最小実装手順。プラグイン自作が要るか、公式の汎用アクション（ホットキー送出／コマンド実行）で済むかの判断。自作する場合の SDK 構成（manifest.json、Node/JS、`.streamDeckPlugin` 配布）。
- ソース境界: Elgato 公式 Stream Deck SDK ドキュメント / Marketplace ガイドライン（一次情報）。
- VOI: 「専用プラグインを作る」か「ドキュメントで公式汎用アクションへ誘導する」かを変える（実装量が桁違い）。
- 反証: 公式の Hotkey/Run アクションでトグルが安定起動できるなら、自作プラグインは不要（VOI低）。

### Q2. 多ボタンマウス／フットスイッチ等の連携
- 目的: Logicool/Razer 等のマウスや汎用フットスイッチで、ボタンに「ホットキー送出」または「コマンド実行」を割り当てる標準手段を確認。
- 出力: ユーザー向け設定手順（各社ソフトでホットキー割当 or `--toggle-record` 実行）。アプリ側に追加実装が要るか。
- ソース境界: 各社公式ヘルプ（Logi Options+ / Razer Synapse）/ OS標準。
- VOI: アプリ側追加実装の要否を変える。
- 反証: 既存ホットキー/CLIで全デバイスが賄えるなら追加実装不要。

### Q3. 類似アプリ（録音/ディクテーション）の物理トリガー実装
- 目的: Talon Voice / Dragon / OBS / Loom / PTT(Push-to-Talk)系が物理トリガーをどう統合しているか（専用プラグイン提供か、ホットキー誘導か、押下中のみ録音=モーメンタリ対応か）。
- 出力: 採用すべきUXパターン（トグル vs モーメンタリ／公式プラグイン提供の有無）。
- ソース境界: 各OSS/製品の公式ドキュメント・リポジトリ。
- VOI: QuickScribeが提供すべき成果物（プラグイン/ドキュメント/モーメンタリ対応）を変える。
- 反証: 主要製品が一様にホットキー誘導のみなら、自前プラグインは過剰。

## Step 8 収束（最重要3問）
Q1（Stream Deck の正道）→ Q3（採用UXパターン）→ Q2（マウス等）。

## メトリクス（GQM）
Goal: 独自実装を避け各デバイス公式手段に橋渡しした物理トリガー体験を最小実装で提供。
Metric: 各連携手段に「公式一次情報で裏取り」「アプリ側追加実装の要否Yes/No」「ユーザー手順が3ステップ以内か」。
