# 原典スナップショット: ISO/IEC/IEEE 29148 / JIS X 0166

- 規格: ISO/IEC/IEEE 29148（Systems and software engineering — Life cycle processes — Requirements engineering）。2011初版 / 2018改訂。
- **JIS X 0166:2014 = ISO/IEC/IEEE 29148:2011 の一致規格(IDT, 翻訳JIS)**。よって29148:2011の内容がJIS X 0166:2014に対応。
- 入手注意: 本文は有料。以下は2011版本文(公開コピー)から確認した短い逐語＋2018版の二次確認。

## 要件と設計の分離（"what" not "how"）— 最重要
Clause 5.2.7 Requirement language criteria（短い逐語）:
> "Requirements should state 'what' is needed, not 'how'."

Clause 5.2.5 "Implementation Free"（2011版、短い逐語）:
> "The requirement states what is required, not how the requirement should be met."

（2018版では "Implementation Free" は廃止され、その趣旨は特性 **"Appropriate"**「…avoids implementation details」に統合。）

## 用語定義（2011 Clause 4.1、短い逐語）
- requirement: "statement which translates or expresses a need and its associated constraints and conditions"
- SRS（software requirements specification）: "structured collection of the requirements … of the software and its external interfaces"
- requirements validation: "… define the right system as intended by the stakeholders"（正しいものを作っているか）
- requirements verification: "… are well formed"（正しく作っているか）

## 階層 Needs → Requirements → Specification → Design
Clause 5.2.1（短い逐語）:
> "The result of requirements engineering is a hierarchy of requirements …"

対応: Needs/Stakeholder intentions → **Stakeholder Requirements(StRS / 業界によりBRS)** → **System Requirements(SyRS)** → **Software Requirements(SRS)** → Design。

## 個別要件の特性
- 2011版: Necessary / Unambiguous / Consistent / Complete / Singular / Feasible / Traceable / Verifiable ＋ Implementation Free。
- 2018版（9特性, 二次確認）: Appropriate / Complete / Conforming / Correct / Feasible / Necessary / Singular / Unambiguous / Verifiable。
- ※JIS X 0166:2014 は2011版ベース（Implementation Freeを含む）。2018版の9特性体系は未反映。

## QuickScribeでの含意
「要件(what)と設計(how)の分離」は著者の主観でなく**国際標準が明文で規定**している。
→ Issue/仕様テンプレで要件セクション(EARS受入基準)と設計セクションを分ける方針([ADR候補])の権威的根拠。
