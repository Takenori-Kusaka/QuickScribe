# ADR-0017: スキーマ版管理と非破壊マイグレーション（設定・エントリ / S5.3・S4.4）

- Status: Accepted
- Date: 2026-06-27
- Deciders: Takenori Kusaka
- 関連: [ADR-0006 スコープ規律], [S4.2 出力形式], [S4.3 タグ]
- 対象 Issue: #38（S5.3 設定スキーマ）, #35（S4.4 データスキーマ版＋migration）

## 背景・課題
1.0品質には「設定やエントリの形式が将来変わっても壊れない／既存データを失わない」土台が要る。現状:
- 設定 = localStorage（多数のキー）＋keyring（秘密）＋メモリ状態。古い/壊れた値で起動が不安定になりうる。
- エントリ = 保管庫の md/txt ファイル。形式変更時に既存を壊さず扱う仕組みが未定義。

## 決定

### 設定（#38）
1. **設定スキーマ版** `settingsVersion` を localStorage に持つ（現行 = 1）。
2. **起動時に検証＋移行**（`validateSettings`）: enum的な値（provider/sttProvider/recordMode/outputFormat/audioFormat/awsAuthMode/refineStyle 等）が不正なら**既定へクランプ**。未知キーは保持（非破壊）。旧版なら移行フックを通して版を更新。
3. 秘密情報は keyring（既存）。検証対象外。

### エントリ（#35）
1. **エントリスキーマ版** をMarkdownフロントマターに `schema: <N>`（現行 = 1）として埋める。txt には付さない（最小形式維持）。
2. **パーサは版の欠落/未知を許容**（legacy=0）。読み取り時に解釈し、**既存ファイルは勝手に書き換えない**（非破壊）。
3. **非破壊保存の不変条件**（既存）: 同名衝突は一意名に退避（上書きしない）／書き込みは一時名→rename。
4. 将来フォーマットを破壊的に変える場合のみ、明示的な移行（読み取り→新版で別名保存、原本は残すか .bak）を別途実装する。本ADRでは**版マーカーと許容パーサ＝移行の土台**までを置く。

## 結果・トレードオフ
- **Pro**: 設定の破損耐性が上がる。エントリは版が刻まれ、将来の非破壊移行が可能に。Obsidian等の外部編集（未知キー追加）も壊さない。
- **Con**: 現時点で実移行は不要（no-op）＝オーバーエンジニアリングを避け、版マーカー＋検証に限定。
- **却下案**: 独自DB/インデックス化 → 現データモデル（プレーンファイル＋localStorage）に過剰。可搬性も損なう。

## 「考えが変わる条件」
- エントリ形式を破壊的に変更する必要が生じたら、本ADRの土台の上に具体的移行（版N→N+1）を追加するADRを起こす。
- 設定が複雑化（ネスト構造化）したら、localStorageから構造化ファイル＋検証スキーマ（JSON Schema等）へ移行を検討。
