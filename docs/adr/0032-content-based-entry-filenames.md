# ADR-0032: エントリのファイル名を「日付＋内容由来ラベル」にする（時刻を廃止）

- Status: Accepted（Deciders 要望 2026-07-24。「文字起こしは本文冒頭・整形はAIタイトルをファイル名に、時刻(-hhMMss)は省略、重複時は index 付加」を明示。同日追記: 名前ソートが日付順になるよう日付を先頭へ）
- Date: 2026-07-24
- Deciders: Takenori Kusaka
- 関連: [ADR-0017 エントリスキーマ](0017-schema-versioning-and-migration.md)（非破壊保存の原則）／ S4.1（保管庫・一意名）／ S4.2（出力形式）

## 背景・課題

保管庫のエントリは `{種別}-{yyyymmdd}-{hhMMss}.{ext}` で命名していたが、タイムスタンプがファイル名の大半を占有し、ファイラや一覧で**内容を見分けられない**。ボイスジャーナルの蓄積は「後から見返す」ことが前提であり（コア価値=思考整理・自己理解）、名前で内容が分かることは発見性に直結する。

## 決定

1. **ファイル名を `{yyyymmdd}-{種別}-{ラベル}.{ext}` にする。時刻(-hhMMss)は含めない。**
   同一日に話し出しが全く同じになるケースは稀であり、時刻よりラベルに文字数を充てる方が見分けやすい。
   **日付を先頭**に置くことで、ファイラの名前ソートがそのまま日付順になる（Deciders 追記 2026-07-24）。
   既存ファイル（旧 `{種別}-{yyyymmdd}…` 形式）はリネームせず、種別推定（`kind_from_filename`）が新旧両形式を判定して後方互換を保つ。
2. **ラベルの由来は種別で変える**:
   - `transcript`（生の文字起こし）/ `note`: **本文冒頭**（サニタイズ後 最大24文字）。
   - `refined`（整形）: **AI 生成タイトル**。整形と同一プロバイダへタイトル生成リクエストを追加送信する（Deciders 選択。API 呼び出しは2回になるがタイトル品質を優先）。生成失敗時は本文冒頭へフォールバックし、**保存は決して止めない**。
   - **レイテンシ設計**: タイトル生成〜保存は整形本文の返却後に**別スレッドで非同期実行**する。コア体験（整形→表示）の待ち時間は従来と不変で、追加のLLM往復はユーザーの目に入らない（第三者レビュー指摘の反映）。副作用として、保存完了直前にアプリを終了すると整形ファイルが残らない極小リスクがあるが、整形結果は画面に表示済みで原文(transcript)も別途保存されるため許容する。
3. **サニタイズ**: Windows 禁止文字(`\/:*?"<>|`)・制御文字を除去、空白列は単一スペースへ、両端のドット/空白を除去（純粋関数 `entry::sanitize_label`）。記号のみ等でラベルが空になる場合は `{yyyymmdd}-{種別}` のみ。
4. **重複時の例外対応**: 同名衝突は既存の一意名付与（`-2`, `-3`…）で吸収する（S4.1 R5 非破壊保存は不変）。
5. **作成時刻は失わない**: md はフロントマター `created`（秒精度 ISO8601）、txt はファイル mtime で従来どおり保持・一覧ソートに使用する（ファイル名から時刻を消しても情報は欠落しない）。

## 結果・トレードオフ

- **Pro**: 一覧・ファイラで内容が一目で分かる。時刻の視覚ノイズが消える。
- **Con/リスク**: (1) 整形保存時に API 呼び出しが1回増える（軽量・冒頭1500文字のみ送信で緩和）。(2) 同日・同一冒頭の衝突が時刻付きより起きやすい（-2 index で吸収、非破壊は不変）。(3) ファイル名から秒単位の作成時刻が読めなくなる（メタデータで保持）。
- **却下案**: (a) 整形出力の見出しからタイトル抽出＝追加コストゼロだがスタイル（逐語等）によっては見出しが無い。(b) 本文冒頭で統一＝「タイトルを考えて付与」の要望を満たさない。

## 検証

- `<yyyymmdd>-transcript-<本文冒頭>.<ext>` / `<yyyymmdd>-refined-<AIタイトル>.md` で保存され、名前ソートが日付順になる。
- 衝突時に `-2` が付き既存ファイルが保持される。タイトル生成失敗でも保存が成功する。
- Windows で不正とならないファイル名のみ生成される（禁止文字・末尾ドット）。
