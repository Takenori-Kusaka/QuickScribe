# ADR-0018: 整形出力言語（翻訳）を専用の永続設定・1パスで提供する（#453）

- Status: Accepted
- Date: 2026-07-01
- Deciders: Takenori Kusaka
- 関連: [ADR-0004 プロダクト位置づけ], [ADR-0006 スコープ規律], [#401 i18n]
- 対象 Issue: #453 / 調査: [docs/research/output-language-translation.md](../research/output-language-translation.md)

## 背景・課題
母語以外で話す/考えるユーザーや、整形メモを他言語話者へ共有する運用では、**話した言語に関わらず読み手に合わせた言語で整形結果を得たい**。一方でコア価値は「ニュアンスを残す整形の知性」であり、翻訳はニュアンス保持と原理的に緊張する。どの設計で提供するかを決める。

## 決定
1. **専用の「出力言語」設定（永続）**として提供する。UI言語（i18nロケール）とは独立。ある作業期間に期待する「出力状態」として保持する（毎回判断ではない）。
2. **既定OFF**。ON時に言語ピッカーを表示（progressive disclosure）。既定言語＝起動時のUI言語。**UI言語に限らず任意言語**（Vietnamese等）を選べる。
3. **1パス**で実行する。整形システムプロンプトに出力言語指示を追記する（`refine.rs`）。**2パス（整形→翻訳の分離）は採用しない**。
4. **原語の文字起こしは常に保持**する。翻訳が効くのは整形出力のみ。
5. 言語定義は **SSOT**（`constants.ts` `LANGUAGES`）で一元管理し、UI言語切替・i18n・出力言語ピッカーが同一定義を参照する。

## 根拠（なぜ）
- 調査（一次情報）で競合（AudioPen/Granola/Audionotes/Otter）が一様に「文字起こし言語」と「出力言語」を分離し、UI言語とも混同していないことを確認。専用設定は定石。
- **2パスの撤回**: クロスリンガル要約研究は2パス優位を示すが、対象は低リソース言語・短文要約であり、QuickScribeの ja↔en・強力な単一LLM・長文には外挿できない（調査でも未検証と留保）。1パスで十分と判断。過度な多段は簡便さ（コア課題「リッチすぎると簡便でなくなる」）も損なう。
- 原語保持により、翻訳品質が不十分でも「本人の声＝原語」を失わない（コア価値の防波堤）。

## 反証可能性（覆る条件）
- ja↔en 長文で1パスがニュアンス・品質を著しく損なうスパイク結果が出れば 2パス（またはオンデマンド再翻訳）へ。
- 想定ユーザーの大半が翻訳を常用するなら既定ONを検討。
