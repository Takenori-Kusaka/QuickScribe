# ADR-0006: スコープ完全性ポリシー（独断のスコープ縮小を禁ずる）

- Status: Accepted
- Date: 2026-06-16
- Deciders: Takenori Kusaka

## Context

本プロジェクトは、ウォーターフォール的に**大きな最終ゴールを先に策定し、余さずやりきる**方針で進める（重厚な開発手順を意図的に採用している）。
にもかかわらず、計画担当（人間/AIエージェント）が「リスクが高い」「重い」「難しい」を理由に、
機能を MVP から外す・Post-MVP送りにする・スコープを暗黙に縮小する、という逸脱が繰り返し発生した
（例: オーディオ入力を「マイクのみ」に縮小、システム音声取り込みを後回し、Stream Deck連携を後回し）。

これは方針違反であり、最終成果物が当初要件を満たさなくなるリスクを生む。

## Decision

**最終ゴールから機能を落とす形のスコープ縮小を、独断で行ってはならない。**

- すべての要件（入力デバイス切替・システム音声取り込み・マイク入力・Stream Deck/マウス等の物理トリガー・
  whisper等エンジン切替・レポート化/整形・Win/Linux配布 等）を、**取りこぼさず**計画・issue化する。
- リスクや難所は「機能を削る」ではなく、**設計・抽象化・分割・段階実装（フェーズ/スパイク）**で対処する。
  - フェーズ分けや技術検証スパイクは可。ただし**最終ゴールの機能集合は不変**とする。
- 提案は「含める/削る」ではなく「**どう全部やりきるか**」の形で行う。
- スコープを変更する必要が生じた場合は、独断で実行せず、**Deciders の明示承認**を得てから新ADRで記録する。

## Consequences

- 計画・issue階層は「全要件網羅」を完了条件とする（[story/task テンプレ](../../.github/ISSUE_TEMPLATE/)と整合）。
- 技術的難所（オーディオ取り込みのWASAPIループバック / PipeWire monitor 等）は第一級サブシステムとして
  正面から設計する（[ADR-0005](0005-tech-stack.md) を全スコープ前提に改訂）。
- 計画フェーズの工数は増えるが、これは意図された重厚プロセスの対価であり受容する。
- このポリシーはエージェントの行動規範でもあるため [CLAUDE.md](../../CLAUDE.md) にも反映する。
