> 個人開発OSS「QuickScribe」（ローカル完結ボイスジャーナル）の設計連載、追章です。前章で私は「文字起こしの床（＝壊れずに末尾まで出る）は死守する」と書き、チャンク化で末尾欠落を根治すると宣言しました。本章はその続きで、**もう一つの床＝「遅すぎては使えない」を実測で殴られ、GPU と Vulkan で実用速度に戻すまで**の記録です。うまくいった話も、**自分が ADR に書いた実測値が10倍間違っていた**という恥ずかしい話も、正直に書きます。コードは v1.3.0 時点。数値は自分の実機での実測と一次情報を脚注で示します。
> リポジトリ: [Takenori-Kusaka/QuickScribe](https://github.com/Takenori-Kusaka/QuickScribe)

## 事件：16分の録音に、3時間

前章で末尾欠落をチャンク化（#600）で根治し、日本語の既定モデルを `large-v3-turbo`（q5量子化）に戻しました。ところが利用者から、**16分42秒の録音に turbo で約180分（3時間）かかった**という報告が来ました。

最初、私は「反復ループの暴走だろう」と疑いました。長い無音で誤フレーズが自己強化し、温度フォールバックの再試行が積み上がって処理時間が桁違いに膨れる ― Whisper 系で知られた挙動です[^loop]。チャンク化＋`no_context` はまさにこの主因（前区間コンテキストの持ち越し）を断つ対策なので、「#600 で直っているはず」と思いました。

**しかし実測は、私の仮説を裏切りました。**

## 解剖：180分は「暴走」ではなく「素の計算量」

自分の実機（Ryzen 7 3700X・AVX2 決定的ビルド）で、同じ録音をチャンク化後の turbo で回し直しました。結果は **196分（RTF≈11.7）**。出力はクリーンで反復もなく、末尾 [16:38] まで到達していました。**末尾欠落は #600 で直っている。でも速度は変わっていない。**

さらに60秒スライスで各モデルを統一条件でベンチしたところ、真実が見えました。

| モデル | エンコーダ | CPU実測RTF | ADR-0025 に書いた値 | 乖離 |
|---|---|---|---|---|
| base | 6層 | 約0.5〜1.95 | 0.13 | 約4〜15倍 |
| kotoba-q5 | 32層 | **14.66** | 0.90 | **約16倍** |
| large-v3-turbo q5 | 32層 | **11.7〜16.3** | 1.15 | **約10〜14倍** |

turbo と kotoba は**同じ32層エンコーダ**を持ちます。両方が RTF≈15。つまり **32層エンコーダ級はこの CPU で RTF≈15 が素の実力**で、180分は暴走やチューニング不足の類ではなく、**物理的な計算量**でした。

そして最大の反省 ― **前章で「実測に基づく」と胸を張って ADR-0025 に記録した RTF 値（base 0.13 / turbo 1.15）は、すべて約10倍以上ずれていました**。おそらく計測方法（debug/release、電源プラン、バックグラウンド負荷）の統制が甘かった。「実測主義」を掲げながら、実測の質を検証していなかったのです。これは設計判断の速度面の根拠が崩れたということで、CERベンチと並ぶ**RTF の統一計測**が要る、という宿題になりました。

> **教訓**：数値には「値」だけでなく「計測条件」を刻む。debug ビルドの whisper は release の桁違いに遅く、電源プランひとつで倍変わる。「実測」と書くなら、統制条件を添えて初めて実測です。

## チャンク化は「品質の床」を直したが「速度の床」は直さない

念のため整理すると、チャンク化（#600）が直したのは**末尾欠落＝品質の床**でした。音声を24秒未満・オーバーラップ付きで分割し、各チャンクを独立に復号することで、sequential seek が末尾を落とす構造問題を回避します。最終チャンクが必ず末尾を含むので、末尾欠落は原理的に消えます。実録音でも末尾 [16:38] 到達を確認済みです。

しかしチャンク化は**総計算量を減らしません**（むしろオーバーラップ分わずかに増える）。CPU 並列（`whisper_full_parallel`）も、whisper.cpp が既に全コアを使うため利得は限定的で、しかもメンテナ自身が「分割点の文字起こしは部分語・文脈喪失で悪化しやすい」と明言しています[^parallel]。**CPU 側のチューニングでは桁は変わらない**、という結論に至りました。

では実用速度（≤1×RT）は CPU では無理なのか。桁を変える手段は、実質ひとつでした ― **GPU**。

## GPU：196分が5.5分になった

配布方針（ADR-0012：単一バイナリ・MIT・多環境CPU）を尊重しつつ、GPU を **opt-in のビルド変種**として足す設計にしました（ADR-0027）。ここで重要な切り分けがあります。

- **CUDA ランタイム**（`cudart` / `cublas`）＝ CUDA EULA Attachment A で**再配布可**。インストーラに同梱すればよい（.NET を self-contained で同梱するのと同じ）[^eula]。
- **NVIDIA ドライバ**（`nvcuda.dll`）＝ 再配布不可・アプリからは同梱できない。これが唯一の外部依存。

whisper-rs の `cuda` feature（vendored sys 0.13.1 に既に存在）でビルドし、自分の RTX 4060 で同じ16分録音を回しました。

| 実行 | 所要 | RTF |
|---|---|---|
| CPU (Ryzen 3700X) | 196.3分 | 11.7 |
| **GPU (RTX 4060, CUDA)** | **5.48分** | **0.33** |

**約36倍**。品質・末尾到達は同等。VRAM 8GB で問題なし。固定費（モデルのGPU転送＋CUDA初期化）が約2.5分あり、チャンクあたりの限界は RTF≈0.17 なので、長い録音ほど速く感じます。「16分に3時間」が「5分半」になった瞬間、これは体験が別物になる、と確信しました。

### 「ドライバが無い人」への正しい振る舞い

GPU 版で悩んだのは、CUDA ランタイムではなく**ドライバの前提**でした。.NET のように「無ければインストーラが連鎖起動」できないか ― 調べた結論は明快でした。**ドライバは再配布不可なので同梱・自動導入はできない。NVIDIA 自身の推奨も「公式DLページへ誘導」＝検出＋案内**です[^drv]。

そこで、利用者が挙げてくれた3基準 ―「①入手先を知らなくても誘導される ②なぜ必要か示す ③不要ならスキップできる」― を満たす形にしました。

- **インストーラ（NSIS フック）**：`nvcuda.dll` を検出し、無ければ **[ドライバ入手ページを開く / このまま続行（CPUで動作）/ 中止]** の3択ダイアログ。サイレントインストールでは「続行」を自動選択。
- **アプリ内**：GPU 不可時は「CPUで実行します」＋ドライバ入手リンク。
- **安全網**：`nvcuda.dll` を遅延ロード（DELAYLOAD）し、GPU 初期化に失敗したら CPU で自動リトライ。**ドライバが無くても起動・動作する**。

「速いから CPU を選ぶ」動機は消え、CPU は**明示的な選択肢から静かな安全網へ**降格しました。

## Vulkan：CUDA を入れなくても、どこまでいけるか

ここで疑問が湧きます。**CUDA はドライバ案内や DLL 同梱の負担が大きい。もっと軽い GPU の道はないか。**

答えが Vulkan でした。whisper.cpp の Vulkan バックエンドは **ベンダー横断（NVIDIA/AMD/Intel）・DLL 同梱不要・必要なのは Vulkan 対応の普通のGPUドライバだけ**[^vulkan]。CUDA のような「専用ドライバ ≥528.33」より、はるかに低い前提です。Ollama も Vulkan バックエンドを提供しています。

自分の RTX 4060（Vulkan 1.4 動作確認済み）で、`whisper-rs` の `vulkan` feature をビルドして実測しました。結論から言うと、**Vulkan は CUDA とほぼ同速**でした。

| 実行 | フル録音(16.7分) | RTF | 備考 |
|---|---|---|---|
| CPU (Ryzen 3700X) | 196.3分 | 11.7 | 実用外 |
| **GPU (CUDA)** | 5.48分 | 0.33 | NVIDIA専用・DLL同梱・ドライバ≥528.33 |
| **GPU (Vulkan)** | **6.10分** | **0.364** | ベンダー横断・DLL同梱不要・普通のGPUドライバのみ |

差はわずか約11%（6.1 対 5.5 分）。RTX 4060 は Vulkan からも fp16・matrix cores 付きで正しく認識されました（`register_device: Vulkan0 = NVIDIA GeForce RTX 4060`）。CUDA を1バイトも同梱せずにこの速度が出ています。**CUDA の重荷（専用ドライバ前提・数百MBのDLL同梱・EULA対応・別インストーラ）を背負わずに実用速度へ届く**ということです。

ひとつ癖があります。**Vulkan は初回にシェーダを実行時コンパイルする固定コスト**が大きく、60秒スライス単体だと RTF≈6（大半がシェーダコンパイル）と遅く見えます。しかしフル録音では償却され RTF 0.364。実利用（数分〜数十分の録音）では誤差です。

### 決定打：「試して落ちたらCPU」は原理的に不可能だった

単一ビルド化の鍵は「**GPU が無い環境で、落ちずに CPU へ行くか**」でした。ここで痛い発見をします。GPU デバイスが無い状態（空ICDで擬似）で whisper に `use_gpu=true` を渡すと、こうなりました。

```
ggml_vulkan: (デバイス0)
fatal runtime error: Rust cannot catch foreign exceptions, aborting
exit code: 0xC0000409 (STATUS_STACK_BUFFER_OVERRUN)
```

**whisper.cpp の GPU 初期化は、使えるデバイスが無いと C++ 例外で abort する。** そして **Rust はこの外来例外を捕捉できない**。私が最初に組んだのは「GPU を試して `Err` が返ったら CPU で再試行」というフォールバックでした。しかし**そもそも `Err` が返ってこない**。プロセスごと落ちるので成立しません。「試してからフォールバック」は幻想でした。

答えは反転します ― **試す前に、確実に知る**。利用者からも鋭い指摘をもらいました。「起動時にマイクやシステム情報を認識するフェーズがあるのだから、そこで GPU の有無を調べればいいだけでは？」と。まさに正鵠を射ています。GPU を触る *前* に、Vulkan ローダの安全な C API で物理デバイス数を数え、**1台以上ある時だけ GPU 経路に入る**（`vkEnumeratePhysicalDevices` は戻り値が `VkResult` で例外を投げない）。デバイス0・ローダ不在・ドライバ不在はすべて即 CPU。これなら abort する経路に一歩も踏み込みません。

実装は Rust から `ash` クレートで約20行です。空ICD（GPU無し擬似）で `gpu_backend_available()` が **abort せず `false` を返す**ことを、実プロセスの統合テストで固定しました。回帰したらテストプロセスが `STATUS_STACK_BUFFER_OVERRUN` で落ちて即バレます。加えて `vulkan-1.dll` を**遅延ロード**（DELAYLOAD）にして、ローダ自体が無い最小 Windows でも EXE が起動できるようにしました。起動→デバイス0検出→CPU、と綺麗に流れます。

## 「CPUオンリーは要らないのでは？」への答え

利用者から鋭い問いが来ました ― 「Vulkan があらゆる環境で動くなら、遅すぎる CPU オンリーは要らないのでは？」

**その直感は正しく、そして実現できました。** Vulkan は「GPU＋Vulkan対応ドライバ」を要り、GPU が無い/ドライバが古い/VM・ヘッドレスといった環境では使えません。でも前節の設計 ― **起動時デバイス検出＋遅延ロード** ― のおかげで、Vulkan ビルドは *そういう環境でも落ちずに CPU で動く*。だから **CPU を「別ビルド・別インストーラ」として持つ必要が消えた**のです。CPU は独立した配布物ではなく、Vulkan ビルドの中の安全網になりました。

| モード | 位置づけ |
|---|---|
| GPU（Vulkan） | **既定・自動選択**（起動時にデバイスを検出し、有れば使う） |
| CPU | **自動フォールバック**（デバイス0・ドライバ無しの時だけ・設定で明示OFFも可） |

結果、配布はこう単純化します ― **単一の Vulkan ビルド（＝1インストーラ・1アップデータ）が、CPU からあらゆるGPUまでを1つでカバーする**。利用者が言った「インストール時点で CPU/GPU を選ばせるなんて、技術事情のUX押し付けだ」という指摘は完全に正しく、CPU/GPU の選択は **インストール時の分岐ではなく、起動時の自動判定＋設定トグル**に落ちました。CUDA は「NVIDIA で最後の1割を絞りたい上級者向けの任意 extra」として温存すれば十分です。

## 学びの整理

- **数値は計測条件とセットで**。「実測に基づく」と書くなら統制条件を添える。ADR-0025 の RTF は10倍間違っていた。
- **床は2枚ある**。品質の床（末尾欠落＝チャンク化で根治）と、速度の床（遅すぎ＝GPUで根治）。片方だけでは実用にならない。
- **GPU の桁の力**。CPU のチューニングでは届かない実用速度（RTF 0.33〜0.36）に、GPU で到達できた。しかも Vulkan は CUDA とほぼ同速でベンダー横断・DLL 不要。
- **「試して失敗→フォールバック」が効かない領域がある**。ネイティブ（C++）が abort する初期化は Rust から捕捉できない。こういう相手には「試す前に安全なAPIで確実に知る」＝**事前検出**に切り替える。これは GPU に限らない設計原則。
- **技術事情をUXに漏らさない**。CPU/GPU はビルドの都合。利用者にインストール時の分岐として見せるのは押し付け。起動時の自動判定＋設定トグルに隠し、**単一インストーラ**へ畳んだ。
- **依存の切り分け**。ランタイム（同梱可）とドライバ（案内のみ）を分けて考えると、順法かつ親切な UX に落ちる。
- **残った宿題**：CERベンチと並ぶ RTF の統一計測基盤（ADR-0025 の10倍ズレの再発防止）、Linux/AMD/Intel 実機での Vulkan 実測。

精度はコモディティだ、と第7章で書きました。速度もまた、ある閾値を割ると「体験そのものが成立しない」という意味で、譲れない床でした。床は、抜けてから気づくものなのかもしれません。
