# CLAUDE.md — QuickScribe

このファイルは Claude Code がこのリポジトリで作業する際の指針。簡潔に保ち、詳細は `docs/` 配下を参照する。

## プロダクトの本質（最優先で守る）

QuickScribe は **思考整理・自己理解のためのローカル完結ボイスジャーナル**。
コア価値は「文字起こし精度」ではなく **「ニュアンスを残しつつ思考を整理する整形の知性」**。
判断に迷ったら [docs/vision.md](docs/vision.md) と [docs/adr/](docs/adr/) に立ち返る。

- 差別化に投資する: 用途特化 / ニュアンス保持 / ローカルプライバシー / 物理ボタン統合体験。
- 差別化にしない: 文字起こし精度・OS対応・ローカル動作そのもの（コモディティ）。
- 設計の核心課題: 「リッチすぎると簡便でなくなる」バランス。機能追加時は常にこれを問う。

## スコープ規律（厳守 — [ADR-0006](docs/adr/0006-scope-completeness-policy.md)）

- **独断でスコープを縮小しない。** 「リスクが高い/重い/難しい」を理由に機能を MVP から外したり Post-MVP送りにしない。
- 全要件を取りこぼさず計画・issue化する。リスクは「削る」のではなく**設計・抽象化・分割・段階実装（フェーズ/スパイク）**で対処する（最終ゴールの機能集合は不変）。
- 提案は「含める/削る」ではなく「**どう全部やりきるか**」の形で行う。スコープ変更は Deciders の明示承認＋新ADRが必要。

## 開発フロー

- ブランチ戦略: **GitHub Flow**。`main` は常にリリース可能。作業は `feat/*` `fix/*` `docs/*` `chore/*` ブランチ。
- レビューが必要な変更（実装・重要設計）は **専用ブランチ + Draft PR**。CI green 後にレビューへ。
- 低リスクな計画/ドキュメントは `main` へ直接コミット可。
- コミットは Conventional Commits（`feat:` `fix:` `docs:` `test:` `refactor:` `chore:`）。
- すべてのコミット末尾に Co-Authored-By トレーラを付与。

## 駆動開発の方針（形だけにしない）

- **仕様駆動**: 機能は仕様書（受入基準＝実行可能な形）から始める。実装より先に「期待される振る舞い」を固定。
- **TDD (t-wada流)**: Red → Green → Refactor を厳守。テストを先に書き、最小実装で通し、設計を磨く。仮実装・三角測量・明白な実装を使い分ける。
- **DDD**: ドメイン（ジャーナリング体験）を中心に。ユビキタス言語を `docs/` で定義し、コードと一致させる。
- 詳細運用は ADR と `docs/process/` に分割記載（コンテキスト肥大回避）。

## 言語・コミュニケーション

- ユーザーへの応答は日本語。コードコメントは英語可。
- 設計判断は ADR に「なぜ」を必ず残す（1決定1ファイル）。

## 作業の進め方

- 大きなタスクは縦切り（Walking Skeleton）で早期に貫通実装し、コア価値を実証してから広げる。
- 非機能（セキュリティ・パフォーマンス・プライバシー）を最初から設計に織り込む。
